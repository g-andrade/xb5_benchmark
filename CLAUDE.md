## Project Overview

`xb5_benchmark` is an Elixir benchmarking suite that compares performance of collection data structures: Erlang's built-in `gb_sets`/`gb_trees` vs. the `xb5` library's `xb5_sets`, `xb5_trees`, and `xb5_bag`. It measures runtime and memory across 50+ operations, 4 build types, and collection sizes from 0 to 15,000 elements.

## Commands

```bash
mix deps.get       # Install dependencies (xb5 is a git dep from github.com/g-andrade/xb5)
mix compile        # Compile
mix format         # Format code (uses styler via .formatter.exs)
mix format --check # Check formatting without modifying
```

**Running benchmarks (via IEx):**
```bash
iex -S mix
```

**For reproducible results (recommended on CPUs with aggressive boost clocks):**

Boost clock variability can cause occasional performance drops, pulling p25 far below the median. To reduce noise:

```bash
# Disable frequency scaling (revert with: cpupower frequency-set -g schedutil)
sudo cpupower frequency-set -g performance

# Pin to a single logical CPU
taskset -c 2 iex -S mix
```

- Avoid CPU 0 (higher OS interrupt activity)
- If SMT/HT is active, pin to one sibling only; check pairs via `/sys/devices/system/cpu/cpuN/topology/thread_siblings_list`

```elixir
# Run full suite, save results to a directory
Xb5Benchmark.run("output/my_run")

# Run specific build types only
Xb5Benchmark.run("output/my_run", build_types: [:sequential, :random])

# Run with keyword filter to target specific operation groups
Xb5Benchmark.run("output/my_run", keywords: ["add", "insert"])

# Post-process: merge all JSON results into a single file
Xb5Benchmark.merge_into_single_json("output/my_run")
```

There are no Mix tasks or test suite — benchmarks are run interactively via IEx.

## Architecture

### Data Flow

1. **`InputStructures`** generates and caches pre-built collection instances at various sizes (`n = 0, 100, 200, ..., 15000`) for each build type. Cached as binary terms in `_cache/input_structures/`.

2. **`Cases`** wraps input structures into benchmark cases, handling case types: bulk constructors, per-iteration ops (with/without keys), and binary set/tree operations with a second collection.

3. **`Groups`** defines parametrized benchmark groups (e.g., `union` generates 12 groups × 3 overlap % × 4 sizes). Group IDs are derived from operation name + params.

4. **`Runner`** executes timing/memory measurements in 60-second batches. Convergence requires 4 consecutive stable measurements per collection size (median variance ≤5%). Additionally, once the total elapsed time exceeds 600 seconds, any Collector that is currently stable (but not yet marked as such) gets force-marked as stable — stopping further measurements for that group. This prevents long-lasting non-convergence that stalls near completion.

5. **`Xb5Benchmark`** (main module) orchestrates everything, captures system info, and writes results as hierarchical JSON files (one per operation), then pivots them into CSV.

### Key Modules

| Module | Path | Role |
|---|---|---|
| `Xb5Benchmark` | `lib/xb5_benchmark.ex` | Main orchestrator, CSV/JSON output |
| `Xb5Benchmark.Runner` | `lib/xb5_benchmark/runner.ex` | Timed measurement loop, convergence logic |
| `Xb5Benchmark.Cases` | `lib/xb5_benchmark/cases.ex` | Benchmark case generation and caching |
| `Xb5Benchmark.Groups` | `lib/xb5_benchmark/groups.ex` | Operation group definitions |
| `Xb5Benchmark.InputStructures` | `lib/xb5_benchmark/input_structures.ex` | Test data generation and caching |
| `Xb5Benchmark.Suite` | `lib/xb5_benchmark/suite.ex` | Macro-generated `run_each_*` operation functions |
| `Xb5Benchmark.Suites` | `lib/xb5_benchmark/suites.ex` | Enum of all suite modules (ErlGbSet, ErlGbTree, ErlXb5Set, ErlXb5Tree, ErlXb5Bag) |
| `Xb5Benchmark.CollectionWrappers.*` | `lib/xb5_benchmark/collection_wrappers/` | Uniform API adapters for each collection impl |

### Build Types

- `:sequential` — Keys inserted in sorted order
- `:random` — Random insertion order (50 variants per size for statistical robustness)
- `:from_ordset_or_orddict` — Pre-sorted keys (tests optimized bulk construction paths)
- `:xb5_adversarial` — Insert 25% extra keys, then delete every 4th (stresses xb5 internal structure). **Currently inactive** — commented out of `all_build_types/0` in `InputStructures` as it was not distinct enough to be worth the extra run time.

### Output Structure

```
output/<run_name>/
├── system_info.json
├── stats_runtime.csv          # Pivoted CSV: rows = (build_type, group, impl), cols = n values
├── stats_memory.csv
├── merged_data.json           # Optional; generated by merge_into_single_json/1
├── report.html                # Optional; generated by generate_report.py
└── build_{type}/{impl_mod}/runtime|memory/{group_id}.json
```

## HTML Report

`generate_report.py` is a standalone Python 3 script that generates a self-contained interactive HTML report from a run's `merged_data.json`.

**Usage:**
```bash
python3 generate_report.py output/my_run
# writes output/my_run/report.html (~3 MB; requires internet for Plotly CDN)
```

**What it shows:**
- Three comparison sections: `xb5_sets` vs `gb_sets`, `xb5_trees` vs `gb_trees`, `xb5_bag` vs `gb_sets`
- Overview table per section: all operations sorted alphabetically within sub-groups, with a proportional speed/memory chip (e.g. `130%` = xb5 is 130% as fast) at the largest measured `n`
- Per-operation detail: line chart (median + p25–p75 band) and a percentage chart anchored at 100% = baseline; parametrised operations (union/intersection/difference/is_disjoint) have overlap % and second-collection-size selectors; named-variant operations (is_equal/is_subset) have variant buttons
- Build type and metric (runtime/memory) selectable from the header; memory charts use dashed lines and a tinted background to distinguish them visually
- Browser back/forward navigation works via the History API

**Key design notes:**
- Runtime measurements are **iterations per second** (higher = faster); memory measurements are **bytes** (lower = better). Color coding and chart orientation reflect this.
- The script strips `merged_data.json` down to only `median`, `p25`, `p75` per measurement point before embedding, reducing the payload from ~30 MB to ~3 MB.
- The `n_values` top-level field in `merged_data.json` is currently `[null]` (a known bug in the Elixir merge code); the script extracts n values from the measurement arrays instead.

### Suite Macro Pattern

`Suite` (via `use Xb5Benchmark.Suite`) generates `run_each_{op}/2` functions and group definitions at compile time using `Module.defines?/2` to conditionally include operations not supported by all collection types. Each suite module (`ErlGbSet`, `ErlXb5Set`, etc.) uses this macro and specifies its `@coll_mod` (the underlying Erlang module).
